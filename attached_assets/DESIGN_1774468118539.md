# Design System Specification: High-Density Technical Precision

## 1. Overview & Creative North Star: "The Synthetic Architect"
This design system is built for the high-stakes, high-velocity world of crypto-infrastructure. It moves beyond the "standard SaaS dashboard" to create an environment that feels like a high-performance terminal. 

**The Creative North Star: The Synthetic Architect.** 
We reject generic fluff in favor of **Technical Brutalism**. The layout is driven by data density and intentional asymmetry. We utilize "Editorial Logic"—where the most critical data points are treated like headlines in a premium financial journal, and the supporting API metadata is tucked into a rigorous, monospaced grid. By overlapping surfaces and utilizing high-contrast tonal shifts, we create a UI that feels engineered, not just designed.

---

## 2. Color Theory & Surface Logic
The palette is rooted in deep obsidian tones, punctuated by high-frequency accents. We prioritize legibility and "Visual Silence" over unnecessary decoration.

### The "No-Line" Rule
To achieve a premium, integrated feel, **direct 1px solid borders are prohibited for sectioning large layout blocks.** Instead, boundaries must be defined by shifts in background tokens (e.g., a `surface-container-low` sidebar against a `surface` main stage). This forces the eye to recognize structural change through mass and tone rather than wireframes.

### Surface Hierarchy & Nesting
Treat the interface as a physical stack of technical components.
- **Base Layer:** `surface` (#121315) – The foundation.
- **Secondary Regions:** `surface-container-low` (#1B1C1E) – Used for navigation or side-panels.
- **Information Nodes:** `surface-container-highest` (#343537) – Used for the most interactive elements like code editors or active trade modules.

### The "Glass & Gradient" Rule
Floating elements (Modals, Command Palettes) must utilize **Glassmorphism**. Apply `surface-variant` at 60% opacity with a `20px` backdrop-blur. 
*Signature Polish:* Main CTAs should use a subtle linear gradient from `primary` (#ADC6FF) to `primary-container` (#4D8EFF) at a 135° angle to provide a "lit from within" tech-glow.

## 3. Typography: The Dual-Engine Scale
We utilize a high-contrast pairing: **Inter** for structural UI and **Geist Mono** for data-heavy strings (wallet addresses, hashes, API keys).

| Level | Token | Font | Size | Intent |
| :--- | :--- | :--- | :--- | :--- |
| **Display** | `display-md` | Inter | 2.75rem | Bold, authoritative market headers. |
| **Headline** | `headline-sm` | Inter | 1.5rem | Section titles; high-impact. |
| **Title** | `title-sm` | Inter | 1.0rem | Standard module labeling. |
| **Body** | `body-md` | Inter | 0.875rem | Default descriptive text. |
| **Code/Data** | `label-md` | Geist Mono | 0.75rem | **Primary for API data & hashes.** |

**Editorial Note:** Always use `label-md` (Mono) for numbers and timestamps to ensure tabular alignment and a "developer-first" aesthetic.

---

## 4. Elevation & Depth: Tonal Layering
We move away from the "shadow-heavy" web. Hierarchy is communicated through the **Layering Principle**.

*   **Natural Lift:** Instead of shadows, place a `surface-container-lowest` card on a `surface-container` background. The slight darkening creates a "recessed" effect that feels more integrated with the hardware.
*   **Ambient Shadows:** For floating menus, use an extra-diffused shadow: `0 24px 48px -12px rgba(0, 0, 0, 0.5)`. The shadow color must never be pure black; it should be a tinted version of `surface-dim`.
*   **The Ghost Border:** If a container requires a border for accessibility, use the `outline-variant` token at **15% opacity**. This "Ghost Border" provides a hint of structure without cluttering the high-density view.

## 5. Component Guidelines

### Buttons (The "Engine" Variant)
*   **Primary:** Gradient fill (`primary` to `primary-container`), `6px` radius, white `on-primary` text. No border.
*   **Secondary:** `surface-container-highest` background with a `ghost-border`. 
*   **Tertiary:** Ghost button. Only a label with `primary` color on hover.

### Input Fields & Technical Arrays
*   **The Input:** Background: `surface-container-lowest`. Radius: `0.25rem`. 
*   **Interaction:** On focus, the border shifts to `primary` at 40% opacity with a subtle `2px` outer glow of the same color.
*   **Data Density:** Use `spacing-2` (0.4rem) between label and input to keep elements tightly coupled.

### Cards & Lists: The "No-Divider" Mandate
**Explicitly forbid the use of horizontal divider lines.** 
*   To separate list items (e.g., transaction history), use a background-color toggle between `surface-container` and `surface-container-low`.
*   Use `spacing-3` (0.6rem) of vertical whitespace to define groups. This creates a "clean-room" environment for complex data.

### Signature Component: The "Terminal Badge"
For status (Success, Error), use `4px` radius badges with a background of the status color at 10% opacity and text at 100% opacity. Example: `Success` badge uses `#22C55E` text on a `#22C55E1A` background.

---

## 6. Do’s and Don’ts

### Do
*   **DO** use monospaced fonts for all alphanumeric data (hashes, amounts, keys).
*   **DO** use asymmetric layouts (e.g., a wide data chart next to a narrow, high-density metric rail).
*   **DO** lean into high-contrast text ratios (`on-surface` for primary, `on-surface-variant` for metadata).

### Don't
*   **DON'T** use 100% opaque borders. They create "visual noise" that fatigues the user in data-heavy environments.
*   **DON'T** use rounded corners larger than `0.5rem (8px)` for cards. Softness dilutes the "technical" feel.
*   **DON'T** use standard drop shadows for buttons or cards; stick to Tonal Layering for a sleeker, more modern aesthetic.

### Accessibility Note
Despite the dark theme, ensure all `label-sm` text maintains a 4.5:1 contrast ratio against its respective surface container. High-end design must never compromise functional clarity.
